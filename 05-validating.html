6  Validating your EBVCube before uploading

By the end of this unit, you will be able to:

  • inspect the structure of your netCDF with ebv_properties() and ebv_datacubepaths();
  • check that the cube values fall in a sensible range with ebv_analyse();
  • confirm the data maps correctly with ebv_map();
  • compare the finished file against the metadata you declared;
  • run a short pre-flight checklist before you upload.

In Chapter 5 you built an EBVCube and gave it a first structural look. This unit goes one step further and checks that the file is right before you upload it to the EBV Data Portal in Chapter 7. Every problem you catch here is cheaper to fix now than after a reviewer finds it.

The checks below use the file you created in the previous unit. Set nc to its path, and use the cube path for the running example, which has no scenario layer:

library(ebvcube)

nc <- file.path(tempdir(), "bird_diversity.nc")   # the file from @sec-creating
dc <- "metric_1/ebv_cube"

6.1 Check the structure

Start with the two functions you already met. ebv_datacubepaths() lists every cube in the file, and ebv_properties() reports the metadata slot by slot. Read them against what you meant to build:

ebv_datacubepaths(nc, verbose = FALSE)

props <- ebv_properties(nc, verbose = FALSE)
props@general$entity_names     # the entities you declared
props@spatial$extent           # c(xmin, xmax, ymin, ymax)
props@temporal$dates           # the time axis

The datacube paths should list every metric and entity you expect, the entity names should read correctly, and the extent and dates should match your data. A missing entity or an empty time axis points to a step that went wrong in creation.

6.2 Check the values

Structure can be right while the numbers are wrong, so read a slice and look at its range. ebv_analyse() returns summary statistics for a cube or a subset:

stats <- ebv_analyse(nc, dc, entity = 1, timestep = 1, verbose = FALSE)
stats[c("min", "max", "mean", "n", "NAs")]

n is the number of valid pixels. If it is zero, or NAs covers the whole grid, the cube never received its data. A min or max far outside what your variable should hold usually means a unit problem or a stray fill value that leaked into the data.

6.3 Check it visually

Numbers only tell you so much. ebv_map() draws a single slice, and one look often catches problems a summary hides, such as data in the wrong hemisphere or an all-empty ocean:

ebv_map(nc, dc, entity = 1, timestep = 1, verbose = FALSE)
If the map looks shifted by 180°

The file is using the [0, 360] longitude convention while the basemap expects [-180, 180]. Read the slice and wrap it before plotting:

r <- ebv_read(nc, dc, entity = 1, timestep = 1, type = "r")
r <- terra::rotate(r)

6.4 Compare against your metadata

Finally, confirm the file says what you told it to say. Read the attributes back and check them against the metadata form (or the ebv_metadata() call) you used to create it:

props@general$title
ebv_properties(nc, dc, verbose = FALSE)@ebv_cube$units

The title, units, entity count, and time coverage should all match your metadata. If a unit or description is wrong, fix it in place with ebv_attribute() from Chapter 5 rather than rebuilding the file.
