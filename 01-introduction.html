2  Introduction

By the end of this unit, you will be able to:

  • define an EBV and explain what it is for;
  • name the six EBV classes and recognize how they appear on the portal;
  • explain why the EBVCube stores data as netCDF, and what its format adds;
  • find and inspect a dataset on the EBV Data Portal.

This unit fills in the concepts you will lean on for the rest of the tutorial: what an EBV is, why the is settled on netCDF, and how datasets are organized. You do not need any software yet. We install ebvcube in Unit 2. For now, everything happens in your browser, on the EBV Data portal.

2.1 What is an Essential Biodiversity Variable?

An Essential Biodiversity Variable (EBV) is a standardized, derived measurement that captures a key dimension of biodiversity and enables consistent monitoring of its status and change over time. The concept was proposed by Pereira et al. (2013) and is maintained by the Group on Earth Observations Biodiversity Observation Network (GEO BON). EBVs are produced by integrating and harmonizing multiple primary data sources, such as field observations, remote-sensing products, and model outputs, into a common, analysis-ready format. As a result, they provide an intermediate layer between raw biodiversity observations and the indicators used for biodiversity assessment, reporting, and decision making.

Think of an EBV as the biodiversity counterpart to an Essential Climate Variable: a small, agreed-upon set of variables that different research groups can produce, compare, and combine because they follow the same definitions. That shared standardization is what makes biodiversity data from different projects, taxa, and regions consistent enough to track over time.

2.2 The six EBV classes

EBVs are organized into six classes, each capturing a different dimension of biodiversity:

  • Genetic composition
  • Species populations
  • Species traits
  • Community composition
  • Ecosystem structure
  • Ecosystem function

The dataset we use throughout this tutorial, Local bird diversity (cSAR/BES-SIM), belongs to the Community composition class.

Canonical classes vs. what you see on the portal

The six EBV classes above are the canonical classification defined by GEO BON. On the EBV Data Portal, you may not see all six listed at once, the portal shows only the classes for which datasets are currently available. You may also come across more specific labels such as Species distribution or Ecosystem disturbance; these are finer sub-categories within the canonical classes.

Ecosystem services is not one of the six EBV classes. It relates to a separate, complementary GEO BON framework, the Essential Ecosystem Service Variables (EESVs), which addresses the benefits people obtain from nature rather than biodiversity itself. For the official EBV classification, refer to the GEO BON definitions page.

2.3 Why netCDF? The EBVCube format

A single EBV has a value at every location (longitude, latitude), at several points in time, and for each biological entity measured (a species, a group, an ecosystem type). Stored as separate map images, that becomes an unwieldy pile of files with no built-in context. The Network Common Data Form (netCDF), long used in the climate sciences, instead holds it all in one self-describing file: the data and the information that explains it units, dates, entities, projection and travel together.

The EBVCube is the standardized, netCDF based data format defined for EBVs. It follows two established conventions: CF v1.8 (how coordinates, units, and time are encoded) and ACDD v1.3 (the attributes portals use to index datasets), plus a few EBV specific rules set out in the EBVCube format specification.

Every data cube inside an EBVCube shares the same four dimensions, think of each pixel as carrying four pieces of information: where (longitude, latitude), when (time), and which entity:

Dimension Meaning
lon Longitude (or projected x-coordinate)
lat Latitude (or projected y-coordinate)
time Time steps (annual, decadal, scenario years, irregular)
entity The biological unit (e.g. species, group of species, or ecosystem type)

The entity dimension allows a single EBVCube to contain data for multiple species or other biological entities. Beyond these four dimensions, metric and scenario groups organize different biodiversity measures or future projections within the same file. We return to that structure in Unit 4.

2.4 Touring the EBV Data Portal

All published EBV datasets are available through the EBV Data Portal. Throughout this tutorial, we will use a single dataset as our running example:

Worked example: Local bird diversity (cSAR/BES-SIM)

  • EBV class: Community composition

Open the dataset page and you will find everything that makes an EBV dataset FAIR (findable, accessible, interoperable, and reusable):

  • a stable dataset ID (1) and a persistent DOI you can cite;
  • a structured JSON metadata record (title, abstract, EBV class, creators, license, entities);
  • the netCDF file itself, available for direct download (≈ 4 MB);
  • an interactive EBV Visualizer for exploring maps and temporal trends without downloading anything.

2.5 Where ebvcube fits

The portal is where EBV data is published and discovered; the ebvcube R package is how you work with it. It sits on both sides of the portal:

  • Reading published data. Give ebvcube a dataset’s numeric ID and it downloads the file, lists the data cubes inside, and reads, subsets, or maps them for you, pulling dataset id = 1 into R takes a couple of lines instead of a manual download and netCDF wrangling.

  • Producing new data. Starting from your own rasters or model output, ebvcube assembles them into a valid EBVCube, ready to upload and publish on the portal with its own DOI.

Read on one side, create on the other: that round trip is the workflow in the Overview diagram, and the path the rest of this tutorial follows, one step at a time.

Checkpoint. You just inspected dataset id = 1 on the portal, its EBV class (Community composition), its DOI, and its file format (netCDF). If those three make sense, you are ready to install the tools in Unit 2.
